## 1. 你要先知道的事

这是一个“Word 模板智能填充”系统，核心目标是：

1. 管理模板（上传、解析、字段配置）
2. 让用户通过对话或导入方式完成字段填充
3. 生成可下载的 Word 文档
4. 在高级链路中支持 AI 文件提取与多引擎兜底

技术方向：

- 后端：.NET 8 + Microsoft Agent Framework + GitHub Copilot SDK
- 前端：Vue3 + TypeScript + Element Plus
- 数据：SQLite + 本地文件存储

## 2. AI 首次进入仓库的阅读顺序

请严格按这个顺序读取：

1. `README.md`：看项目定位、启动方式、文档入口
2. `01_kb_系统功能详解.md`：看功能边界和核心流程
3. `02_ref_项目进度与变更记录.md`：看阶段状态与历史上下文
4. `01_kb_需求定义_mvp.md`：看需求原始约束和 MVP 边界
5. `03_guide_项目执行计划.md`：看里程碑和交付路径
6. `W*_实现流程.md` + `W*_验收报告.md`：定位某周实现细节和验收依据

如果用户问“这个系统实现了什么”，优先基于 `01_kb_系统功能详解.md` 回答，而不是直接从零散代码推断。

## 3. 代码结构速览（高频）

后端主目录：`backend/FrameAgentWordFill`

- `Controllers`：API 入口（Chat、Import、Generate、Templates）
- `Services`：业务编排（对话、导入、生成、多引擎）
- `Repositories`：SQLite 数据访问
- `Tools`：解析、匹配、生成等工具类
- `Models`：领域模型（Templates/Chat/Import/AIExtraction 等）
- `Data/DatabaseInitializer.cs`：数据库表初始化

前端主目录：`frontend/src`

- `views`：页面（Home、TemplateManager、TemplateUpload、ChatFill、ImportFill）
- `router/index.ts`：路由入口
- `App.vue`：主布局与导航

## 4. 关键业务链路（必须理解）

### 4.1 模板管理链路

上传 Word 模板 -> 解析占位符/表格字段 -> 存储模板与字段配置 -> 启用模板。

### 4.2 AI 对话填充链路（W4）

启动会话 -> 用户自然语言输入 -> AI 提取字段 -> 流式返回 -> 会话持久化 -> 生成文档。

### 4.3 导入填充链路（W5）

上传文件（Excel/JSON/Word） -> 字段匹配（精确/模糊/语义） -> 人工确认 -> 生成文档。

### 4.4 AI 文件提取链路（W7）

上传参考文件（PDF/图片等） -> 内容解析 -> AI 批量提取 -> 置信度分级 -> 人工确认 -> 生成文档。

## 5. AI 回答用户时的优先原则

1. 不把 README 当作实现细节文档。
2. 功能盘点优先引用“功能文档 + 验收文档”，再做代码交叉验证。
3. 涉及“是否已完成”，要给出阶段结论（W1-W8）并标注“阶段性通过/已完成/进行中”。
4. 涉及“可否上线”，默认提醒：仍需端到端回归和环境配置核验。

## 6. 常见误区与纠偏

1. 误区：把所有设计细节继续堆进 README。  
   纠偏：README 只保留入口导航，细节写到专题文档。

2. 误区：仅凭某个 Service 文件就判断全系统可用。  
   纠偏：至少交叉看 Controller、Repository、前端页面和验收报告。

3. 误区：忽略多引擎兜底和降级路径。  
   纠偏：回答 AI 能力时必须说明主引擎、备用引擎和降级策略。

4. 误区：把“文档计划”当“代码已实现”。  
   纠偏：明确区分“计划项”“已实现项”“阶段验收项”。

## 7. 【重要】当用户报告新问题时的处理流程

当用户说"我遇到一个问题"时，按以下流程处理：

### 7.1 第一步：问题分类与信息收集

**立即追问这 5 个问题**（帮助定位）：

1. **影响范围**：这个问题影响哪个功能模块？（模板管理 / 对话填充 / 导入 / AI 提取 / 文档生成）
2. **现象描述**：具体表现是什么？（字段无法识别 / 报错信息 / 功能不工作 / 性能下降）
3. **重现步骤**：如何稳定重现？（附带具体操作步骤和测试数据）
4. **环境信息**：运行环境如何？（Windows / 本地开发 / 日期 / 模板文件样本）
5. **已尝试**：用户是否已做过什么排查？（日志检查 / 重启 / 其他尝试）

### 7.2 第二步：快速查阅知识索引

收集到问题信息后，**按优先级查阅以下文档**：

| 优先级 | 查阅文档 | 目的 |
|--------|---------|------|
| 1️⃣ | `01_kb_系统功能详解.md` | 确认功能是否应该被支持 |
| 2️⃣ | `02_ref_项目进度与变更记录.md` | 检查功能是否已实现（W1-W8） |
| 3️⃣ | 对应的 `W*_验收报告.md` | 查阅验收标准和已知限制 |
| 4️⃣ | `11_module_功能问题排查手册.md` | 查阅已知问题的排查方法 |
| 5️⃣ | `06_issue_*.md`（如果存在） | 查阅是否已有相同问题的需求文档 |

**查阅逻辑示例**：
```
用户问题：模板上传后字段无法识别
  ↓
查 01_kb → 确认"字段自动解析"是 W2 功能
  ↓
查 02_ref → 确认 W2 已完成
  ↓
查 W2_验收报告 → 查看"占位符规范化"的支持范围
  ↓
查 11_module_排查手册 → 查看"字段无法识别"的排查方案
  ↓
查 06_issue → 查看是否有类似的已报告问题
```

### 7.3 第三步：问题诊断与根因分析

**按以下顺序诊断**：

#### 3.1 确认是否属于**已知限制**

查阅验收报告的"⚠️ 缺失的功能"或"👉 已知限制"部分。

**示例**：
- W2 验收报告中记录：不支持旧版表单域、书签、内容控制
- 若用户模板使用了这些格式 → 问题找到，属于功能缺失，无需代码调查

#### 3.2 确认是否属于**已报告的 Issue**

查阅 `06_issue_*.md` 系列文档，搜索关键词。

**示例**：
- 若问题是"字段无法精确识别"
- 查 `06_issue_w8_模板解析精确识别.md` 可找到完整的需求和改进方案

#### 3.3 确认是否属于**使用错误**

查阅排查手册 `11_module_功能问题排查手册.md`，按检查清单排查。

**示例检查项**：
- 模板格式是否正确（.docx 还是 .doc）
- 占位符格式是否符合规范（`{字段名}` 而非 `$字段名$`）
- 字段名是否包含非法字符
- 数据库是否已初始化

#### 3.4 确认是否属于**新的 Bug**

上述 3 步都查不到 → 这是一个新发现的 Bug，需要创建 Issue 文档。

### 7.4 第四步：与用户沟通诊断结果

**按以下方式反馈给用户**：

#### 情况A：属于已知限制

## 8. 【W8补充】模板解析方法论（占位符与点号字段）

当用户反馈“模板字段识别不完整”时，按以下方法处理：

### 8.1 占位符语法兼容

解析策略应覆盖：

1. `{字段名}`
2. `{{字段名}}`
3. `[字段名]`
4. `(字段名)`

执行原则：

1. 不因语法差异直接失败。
2. 先识别原始语法，再统一规范化为内部标准（建议 `{字段名}`）。
3. 规范化过程写入 warnings，前端可回显给用户。

### 8.2 点号字段兼容

为避免 `{Acronym.WordAcronym}` 漏识别，点号规则应调整为：

1. 只要包含 `.` 且段数 >= 2，就作为表格/复合字段候选。
2. 两段：`A.B` -> 表名 `A`，列名 `B`。
3. 多段：`A.B.C` -> 表名 `A`，列路径 `B.C`。
4. 无法明确归类时，不静默丢弃，写 warning 并保留候选。

### 8.3 AI 双轨策略

模板上传建议链路：

1. 本地规则解析（主）
2. AI 语义解析（辅）
3. 差异比对
4. warnings 输出
5. 按本地解析结果入库

说明：AI 失败、超时或格式异常不应阻断上传。

### 8.4 推荐引用文档

1. `docs/06_issue_w8_模板上传AI双轨解析比对.md`
2. `docs/05_api_接口规范.md`
3. `AI_QUICK_START.md`

## 9. AI 修改本仓库文档时的规则

1. README 只做入口，不写长篇实现细节。
2. 功能说明更新到 `01_kb_系统功能详解.md`。
3. 里程碑/版本变化更新到 `02_ref_项目进度与变更记录.md`。
4. 周级实现细节写到对应 `W*_实现流程.md`。
5. 验收结论写到对应 `W*_验收报告.md`。
6. **新问题或 Bug 创建到 `06_issue_wN_*.md`**（遵循问题文档规范）。

## 10. 给 AI 的输出模板（建议）

当用户问"系统做了什么"时，可按这 4 段输出：

1. 当前阶段结论（例如：W7 已阶段性通过，W8 收尾中）
2. 已实现能力清单（模板、生成、对话、导入、AI 提取）
3. 技术与数据架构简述（后端/前端/存储）
4. 风险或待确认项（环境、回归、验收）

## 11. 快速核对清单（AI 自检）

在给用户“完成度”结论前，先自检：

- 是否已读 `01_kb_系统功能详解.md`
- 是否已读 `02_ref_项目进度与变更记录.md`
- 是否区分“已实现”和“计划中的描述”
- 是否给出阶段状态而非绝对承诺

满足以上 4 条，再输出最终判断。
