# 通用Word智能填充Agent - 项目执行计划

**文档版本**：V1.0  
**制定日期**：2026年05月14日  
**项目周期**：8周  
**项目性质**：实验性 MVP 项目

---

## 📋 项目概览

### 目标
构建一个基于 .NET + Microsoft Agent Framework 的 Word 智能填充系统，支持对话式填充、导入填充和 AI 智能填充三种模式，实现单文档生成（最多同时5条）的 MVP 验证。

### 技术栈
- **后端**：C# + .NET 8 + Microsoft Agent Framework + GitHub Copilot SDK
- **前端**：Vue3 + TypeScript + Element Plus
- **文档引擎**：OpenXML SDK
- **数据存储**：SQLite
- **AI 能力**：GitHub Copilot SDK（主） + Azure OpenAI（备）

### MVP 范围
✅ 单文档生成（≤5条）  
✅ 对话引导填充 + 快捷指令  
✅ 外部数据导入（Excel/JSON/Word）  
✅ 表格字段填充  
✅ AI 智能填充（文件提取 + 置信度确认）  
❌ 批量处理（>5条）  
❌ 运营统计  
❌ 复杂权限系统

#### 验收标准
```bash
# 用户上传一个项目申请 PDF：
1. 系统解析 PDF 内容（文本+表格+图片 OCR）
2. AI 调用 LLM 批量提取字段：
   "请从以下文档中提取项目申请表的字段：项目名称、负责人、预算、团队成员..."
3. 返回结果：
   - {项目名称}: "智能文档系统"（置信度: 95%，匹配方式: 精确匹配）
   - {负责人}: "张三"（置信度: 90%，匹配方式: 语义匹配 ← "项目负责人"）
   - {预算}: "50万元"（置信度: 60%，匹配方式: 模糊匹配） ← 需人工确认
   - {团队成员}: [李四, 王五]（置信度: 85%，匹配方式: 表格提取）
4. 用户在确认界面查看：
   - 高置信度字段（绿色）：可直接确认
   - 中置信度字段（黄色）：建议人工检查
   - 低置信度字段（红色）：必须人工修改
5. 用户批量确认或修改后，一键生成文档
```

#### 技术要点
- **复用 W4 的 Agent 和 LLM 能力**，避免重复集成
- **三层匹配策略**：
  1. 精确匹配：文档字段名与模板字段名完全一致
  2. 模糊匹配：使用编辑距离算法（如 "项目名" → "项目名称"）
  3. 语义匹配：使用 LLM 理解语义（如 "项目负责人" → "负责人"）
- **置信度计算**：综合匹配方式、LLM 输出质量、数据格式校验

#### 技术风险
⚠️ **LLM 调用成本**：批量提取可能消耗大量 Token  
   - **缓解措施**：优化 Prompt，分批提取，使用缓存

⚠️ **文件解析复杂度**：复杂 PDF（扫描件、多列布局）解析困难  
   - **缓解措施**：使用 OCR + 布局分析，给出解析质量提示

⚠️ **AI 提取准确率**：复杂文档的提取准确率可能不理想  
   - **缓解措施**：提供置信度评分和友好的人工确认界面

## ⚠️ 风险管理

### 高风险项
1. **LLM 集成和稳定性**（W1, W4, W7）
   - **影响**：AI 对话和提取功能是核心能力，LLM 不可用将导致项目无法交付
   - **应对**：
     - W1 提前集成并测试 LLM 连接
     - 实现多引擎兜底（Copilot SDK → Azure OpenAI → OpenAI API）
     - 准备降级方案（规则匹配 + 人工确认）

2. **Word 文档解析复杂度**（W2-W3）
   - **影响**：可能导致模板解析不准确，影响核心功能
   - **应对**：提供模板预检工具，引导用户使用规范模板

3. **AI 对话理解准确率**（W4）
   - **影响**：AI 可能无法准确理解用户意图或提取字段
   - **应对**：
     - 精心设计 Prompt 和示例
     - 提供字段预览和修改功能
     - 支持快捷指令降低理解难度

4. **表格填充边界情况**（W3）
   - **影响**：复杂表格场景可能填充失败
   - **应对**：MVP 先支持简单表格，复杂场景给出提示

### 中风险项
1. **前后端联调效率**（W4-W5）
   - **影响**：可能导致进度延误
   - **应对**：
     - 提前定义 API 契约（特别是流式接口）
     - 使用 Mock 数据并行开发
     - W4 优先打通 AI 对话主链路

2. **AI 文件提取准确率**（W7）
   - **影响**：用户可能需要大量手动修正
   - **应对**：提供置信度评分和友好的确认界面

3. **LLM 调用成本**（W4-W7）
   - **影响**：频繁调用 LLM 可能导致高额成本
   - **应对**：
     - 优化 Prompt 减少 Token 消耗
     - 实现会话缓存和结果缓存
     - 使用成本较低的模型（如 GPT-3.5）做基础对话
