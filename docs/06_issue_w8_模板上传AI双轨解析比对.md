# Issue: 模板上传增加 AI 双轨解析比对（W8）

创建日期：2026-05-14  
最后更新：2026-05-14（补充：占位符语法扩展与点号字段兼容）  
优先级：P1  
状态：待实施

---

## 1. 需求背景

当前模板上传链路仅执行本地解析（OpenXML + 正则），缺少“语义侧”校验。

现状链路：

- 上传模板 -> 本地解析占位符/表格 -> 保存模板

问题：

- 对复杂格式、业务语义字段、模糊命名场景，无法在上传时给出“结构是否合理”的二次提示。
- 上传成功后才在后续流程中暴露字段问题，排查成本高。
- 当前占位符语法兼容性不足，主要识别 `{}`，对 `{{}}`、`[]`、`()` 这类自定义写法支持不完整。
- 当前表格字段点号规则过于严格，存在 `{Acronym.WordAcronym}` 等命名形式识别失败风险。

目标：

- 在模板上传时增加 AI 解析作为第二轨道。
- 对比“本地解析结果”与“AI 解析结果”的差异。
- 将差异写入警告（warnings），不阻断上传。
- 扩展占位符语法识别能力，并统一规范化为内部标准格式。
- 修复点号字段识别边界，确保 `{Acronym.WordAcronym}` 等表格参数稳定可识别。

---

## 2. 功能目标与边界

### 2.1 功能目标

1. 本地解析仍为主结果来源（主链路不变）。
2. AI 解析作为增强校验（可开关、可降级）。
3. 差异结果统一进入 `parseResult.warnings`。
4. 上传接口返回 AI 比对摘要，便于前端展示。
5. 支持多种占位符语法：`{}`、`{{}}`、`[]`、`()`（按配置启用）。
6. 支持点号字段命名兼容：至少支持 `{A.B}`，并兼容 `{A.B.C}` 等扩展写法。

### 2.2 非目标（本期不做）

1. 不以 AI 结果直接覆盖数据库字段配置。
2. 不因 AI 调用失败而中断上传。
3. 不做复杂自动修复（如自动重命名字段）。
4. 不在本期引入完整表达式语法（如函数调用、条件语法）。

---

## 3. 占位符语法扩展需求

### 3.1 支持语法

本期要求解析器支持以下占位符写法：

1. 单花括号：`{字段名}`
2. 双花括号：`{{字段名}}`
3. 中括号：`[字段名]`
4. 小括号：`(字段名)`

约束：

1. 同一模板允许混用，但内部应规范化为统一标准（建议统一为 `{字段名}`）。
2. 非标准写法不报错中断，写入 `warnings` 并提示“已规范化”。

### 3.2 点号字段命名规则

为解决 `{Acronym.WordAcronym}` 识别问题，点号规则改为：

1. 只要占位符中包含点号且段数 >= 2，即视为“表格/复合字段候选”。
2. 兼容两段写法：`{A.B}`（A=表名，B=列名）。
3. 兼容多段写法：`{A.B.C}`（建议解析为 A=表名，`B.C`=列路径）。
4. 对无法明确归类的点号字段，保留为候选并写入 `warnings`，避免直接丢弃。

### 3.3 规范化策略

建议新增“语法标准化”步骤：

1. 先识别原始包裹符号（`{}`、`{{}}`、`[]`、`()`）。
2. 统一转换为内部标准 token（如 `{...}`）。
3. 记录来源语法与规范化说明，便于回显和排查。

## 4. 设计方案

### 3.1 接口层设计

接口：`POST /api/templates`

新增参数：

- `aiVerify`（可选，bool，默认 `false`）

返回增强：

- `parseResult.warnings` 追加 AI 比对差异提示。
- 增加 `aiVerification` 摘要（是否启用、是否可用、差异数量）。

说明：默认关闭 AI 比对，确保与现有调用兼容。

### 3.2 服务层编排

建议在 `TemplateService.UploadTemplateAsync` 增加分支：

1. 执行本地解析，得到 `localResult`。
2. 当 `aiVerify=true` 时调用 AI 模板解析器，得到 `aiResult`。
3. 执行差异比较器，得到 `diffWarnings`。
4. 将 `diffWarnings` 合并到 `localResult.Warnings`。
5. 保存模板字段仍以 `localResult` 为准。

建议新增组件：

- `TemplateAiParser`：负责 AI 提示词构建 + 结果反序列化。
- `TemplateParseComparer`：负责字段/表格差异检测与告警文本生成。
- `TemplatePlaceholderNormalizer`：负责占位符语法识别与统一规范化。

## 5. AI 提示词设计（业务版）

建议输入：

1. 模板原始文本片段（段落文本 + 表格表头文本）。
2. 业务上下文（合同/申报/审批等模板类型）。
3. 占位符规则说明（例如标准占位符格式）。

建议输出格式（严格 JSON）：

```json
{
  "fields": [
    { "name": "项目名称", "type": "text", "required": false }
  ],
  "tables": [
    {
      "name": "成员清单",
      "columns": ["姓名", "岗位", "电话"],
      "rowType": "dynamic"
    }
  ],
  "notes": ["字段‘联系电话’建议类型为 phone"]
}
```

建议提示词模板：

```text
你是模板结构分析助手。请根据给定的 Word 模板文本内容，识别业务填充字段与表格结构。

要求：
1) 仅返回 JSON，不要输出额外解释。
2) 字段名必须使用模板中的中文原词，不做同义词改写。
3) fields 仅包含非表格字段。
4) tables 中每个对象包含 name、columns、rowType(dynamic/fixed)。
5) 识别多种占位符包裹语法：{}、{{}}、[]、()。
6) 点号字段（如 Acronym.WordAcronym）优先按表格字段处理。
7) 如果无法确定，请保守输出并在 notes 说明。

模板文本如下：
{TEMPLATE_TEXT}

业务场景：
{BUSINESS_CONTEXT}
```

## 6. 差异比对规则

### 5.1 字段差异

1. 本地有、AI 无：输出“可能漏识别”警告。
2. AI 有、本地无：输出“可能漏解析”警告。
3. 同名字段类型不一致：输出“类型建议冲突”警告。
4. 仅语法不同但语义相同（如 `{{项目名称}}` vs `{项目名称}`）：标记为“语法已规范化”，不算结构冲突。

### 5.2 表格差异

1. 表格名仅单边存在：输出“表格识别不一致”警告。
2. 同表列数量不一致：输出“列数量不一致”警告。
3. 同列名差异：输出“列定义不一致”警告。
4. 点号字段段数不一致（如 `A.B` vs `A.B.C`）：输出“列路径层级不一致”警告。

### 5.3 告警分级建议

1. `info`：仅语义建议，不影响流程。
2. `warning`：结构不一致，建议人工确认。
3. `critical`：主干字段大量缺失（例如缺失率 > 30%），建议退回模板检查。

本期可先全部落入 `warnings` 文本，不强制结构化分级。

---

## 7. 降级与容错策略

1. AI 服务不可用：记录日志并追加警告 `AI 比对未执行，已降级为本地解析`。
2. AI 输出非 JSON：记录日志并追加警告 `AI 返回格式异常，已忽略 AI 比对结果`。
3. AI 超时：建议超时 8~12 秒，超时后直接降级。
4. 任何 AI 异常不影响上传成功与模板保存。
5. 自定义语法无法识别时，不中断上传，写入警告并保留原始 token 供人工核对。

## 9. 验收标准

1. 上传模板时，`aiVerify=false` 行为与当前一致。
2. 上传模板时，`aiVerify=true` 且 AI 正常：
   - 返回包含差异告警。
   - 模板保存成功。
3. `aiVerify=true` 且 AI 不可用：
   - 上传仍成功。
   - 返回降级告警。
4. 同一模板在前端可看到本地解析警告 + AI 比对警告。
5. 以下写法均可识别为同一字段：`{项目名称}`、`{{项目名称}}`、`[项目名称]`、`(项目名称)`。
6. `{Acronym.WordAcronym}` 可被识别为有效表格/复合字段，不再丢失。
7. `{A.B.C}` 等多段点号字段可进入候选结果并产出明确 warning 或标准化结果。

## 10. 测试用例建议

1. 基础模板（字段清晰）：预期无差异或低差异。
2. 含复杂表格模板：预期能看到表格列差异提示。
3. 故意写模糊字段名模板：预期能看到语义建议警告。
4. 模拟 AI 断连：预期上传成功且有降级警告。
5. 四种语法混合模板：预期字段总数正确，warnings 包含“语法规范化”提示。
6. 包含 `{Acronym.WordAcronym}` 的表格模板：预期识别成功并可回显。
7. 包含 `{A.B.C}` 的模板：预期可识别为扩展点号字段并有层级提示。

## 11. 风险与注意事项

1. AI 结果不稳定：必须固定 JSON 输出并做严格解析容错。
2. 上传时延增加：建议默认关闭，仅在需要时开启。
3. 成本控制：建议限制模板文本截断长度（例如 6k~10k 字符）。
4. 隐私与合规：若模板含敏感信息，需做脱敏或租户隔离策略。
5. 语法放宽后误识别风险上升：需增加黑名单规则与最小长度约束。

---

## 12. 实施顺序建议

1. 先实现“本地 + AI + 警告合并”（不改数据库）。
2. 增加占位符规范化器（支持 `{{}}`、`[]`、`()`）与点号字段修订。
3. 再补充 `aiVerification` 响应摘要。
4. 最后做前端告警展示优化与灰度开关。

预计工作量：3~5 人日（含联调与回归）。
