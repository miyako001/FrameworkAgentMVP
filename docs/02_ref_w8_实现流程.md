# W8 实现流程 - 验收与收尾

**周期**：第 8 周  
**里程碑**：M8 - 验收与收尾  
**目标**：完成端到端回归、问题修复、文档收尾与验收发布准备

**⚠️ 核心功能说明**：
- 本周是项目的**交付闭环周**，重点不是新增功能，而是确保已有能力可稳定上线
- 覆盖三条核心链路：**AI 对话填充**、**导入填充**、**AI 文件智能提取**
- 建立从测试发现问题到修复验证再到文档沉淀的闭环
- 输出可验收、可演示、可复现的最终交付资产

## 步骤 1：冻结范围并建立 W8 验收基线

### 1.1 范围冻结（Scope Freeze）

执行原则：
- W8 不再新增需求，仅处理已验收范围内问题
- 新增需求统一登记到后续版本 Backlog
- 仅允许以下类型变更进入 W8：
  - 阻塞上线的 Bug 修复
  - 明确的性能瓶颈优化
  - 错误提示和可观测性补强

### 1.2 建立验收基线清单

基线清单（与 `03_guide_项目执行计划.md` 对齐）：
- 管理员可上传模板并完成字段配置
- 用户可通过 AI 对话方式完成字段填充并生成文档
- 用户可通过导入模式（Excel/JSON/Word）完成字段匹配与生成
- 用户可通过 AI 文件模式上传参考文件并提取字段
- 系统可正确生成普通字段和表格字段
- 并发生成能力满足 MVP 限制（最多 5 条）
- 错误处理与日志覆盖核心场景

### 1.3 建立缺陷与回归台账

建议字段：
- `IssueId`
- `模块`
- `严重级别`（P0/P1/P2）
- `复现步骤`
- `预期结果`
- `实际结果`
- `责任人`
- `状态`（Open/Fixing/Resolved/Verified/Closed）
- `回归结果`

## 步骤 2：执行端到端测试（E2E）

### 2.1 对话模式回归（W4 主链路）

验证范围：
- 创建会话与恢复会话
- 流式输出（SSE）稳定性
- 字段提取准确性（单字段/多字段）
- 快捷指令识别（如一次性填充、修改字段）
- 文档生成结果正确性

关键接口：
- `POST /api/chat/start`
- `POST /api/chat/message`
- `GET /api/chat/stream/{sessionId}`
- `POST /api/generate`

验收标准：
- 主链路成功率 >= 95%
- 对话中断可恢复
- 生成文档与输入字段一致

### 2.2 导入模式回归（W5 主链路）

验证范围：
- Excel/JSON/Word 三类文件导入
- 自动字段匹配与置信度展示
- 人工调整映射后再次生成
- 缺失字段补全流程

关键接口：
- `POST /api/import/upload`
- `POST /api/import/match`
- `POST /api/import/confirm`
- `POST /api/generate`

验收标准：
- 常见模板自动匹配准确率 >= 80%
- 人工修正流程无阻塞
- 导入失败场景有明确可操作提示

### 2.3 AI 文件模式回归（W7 主链路）

验证范围：
- PDF/Word/Excel/图片 文件上传与解析
- 多引擎兜底链路（Copilot -> Azure OpenAI -> OpenAI -> 规则）
- 置信度分级展示（高/中/低）
- 人工确认后生成文档

关键接口（按现有控制器实际路由调整）：
- `POST /api/import/ai-extract`
- `POST /api/import/ai-confirm`
- `POST /api/generate`

验收标准：
- AI 提取整体可用，失败时可回退
- 低置信度字段必须可人工覆盖
- 端到端生成成功率 >= 90%

## 步骤 3：缺陷分级与修复闭环

### 3.1 缺陷分级规则

- **P0（阻塞）**：核心链路不可用、数据错误、文档生成失败
- **P1（高优）**：高频失败、明显体验断层、关键提示缺失
- **P2（中低）**：低频边界问题、文案或交互优化项

处理 SLA 建议：
- P0：当日修复并回归
- P1：48 小时内修复并回归
- P2：W8 内评估修复或登记后续版本

### 3.2 修复流程

```
发现问题 -> 复现并归档 -> 评审优先级 -> 指派责任人
-> 修复并自测 -> 提交回归 -> 测试复核 -> 关闭问题
```

### 3.3 修复验证重点

- 不引入跨模块回归
- 同类问题批量排查（避免只修一个点）
- 所有 P0/P1 必须有回归记录

## 步骤 4：性能优化与异常恢复验证

### 4.1 性能验证场景

- 并发 5 条文档生成请求
- 大文件导入（接近上限）
- 长会话连续消息交互
- AI 提取中多引擎切换场景

建议指标：
- 生成接口 P95 响应时间 <= 8s（参考目标，可按实测调整）
- 上传解析接口 P95 响应时间 <= 10s
- 关键 API 错误率 <= 3%

### 4.2 异常恢复验证

覆盖场景：
- Copilot 不可用时的自动降级
- 上游模型超时/限流重试
- 非法文件类型上传
- 模板字段缺失或结构异常

验收标准：
- 异常场景均有结构化错误码与用户可理解提示
- 系统不崩溃，且可继续处理下一请求

## 步骤 5：完善日志、监控与错误提示

### 5.1 后端日志规范化

建议在以下节点统一记录：
- 会话创建/恢复
- 文件上传与解析耗时
- AI 调用开始/结束/降级切换
- 文档生成开始/完成/失败

推荐日志字段：
- `TraceId`
- `SessionId`
- `TemplateId`
- `Engine`
- `ElapsedMs`
- `ErrorCode`

### 5.2 错误提示优化

原则：
- 面向用户：说明原因 + 下一步建议
- 面向运维：保留可检索错误码和上下文
- 面向开发：日志可定位到控制器/服务层

示例：
- 用户提示：`当前 AI 引擎繁忙，已自动切换备用引擎，请重试。`
- 日志记录：`AI_ENGINE_FALLBACK | Engine=Copilot -> AzureOpenAI | TraceId=...`
