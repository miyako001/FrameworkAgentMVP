# AI 工作流指令（.instructions.md）

## 当用户报告问题时的自动工作流

**触发条件**：用户说"我遇到问题"、"这有个Bug"、"我发现一个问题"等

**自动执行流程**：按照 `docs/03_guide_ai协作与项目理解指南.md` 中的**第 7 章问题处理流程**执行

---

## 工作流步骤（自动化清单）

### Step 1: 问题信息收集（自动追问）

如果用户描述不完整，立即追问这 5 个问题：

```
问题分类表单：
├─ 【影响范围】这个问题影响哪个功能模块？
│   （模板管理 / 对话填充 / 导入 / AI 提取 / 文档生成）
├─ 【现象描述】具体表现是什么？
│   （字段无法识别 / 报错信息 / 功能不工作 / 性能下降）
├─ 【重现步骤】如何稳定重现？
│   （附带具体操作步骤和测试数据）
├─ 【环境信息】运行环境如何？
│   （Windows / 本地开发 / 日期 / 模板文件样本）
└─ 【已尝试】用户是否已做过什么排查？
    （日志检查 / 重启 / 其他尝试）
```

**AI 行为**：如果上述信息不足，自动生成一个表单并回复用户：

```
根据你的描述，我需要收集更多信息来帮助定位问题。请按以下表单填写：

【影响范围】

- [ ] 对话填充（Chat 功能）
- [ ] 导入填充（Import 功能）
- [ ] AI 提取（文件提取功能）
- [ ] 文档生成（Generate 功能）
- [ ] 其他

【现象描述】
[请描述具体现象]

【重现步骤】
1. 
2. 
3. 

【环境信息】
- 操作系统：
- .NET 版本：
- 其他：

【已尝试】
- [ ] 检查日志
- [ ] 重启服务
- [ ] 其他：
```

### Step 2: 知识索引快速查阅（自动执行）

**按优先级查阅**（不需要用户重复说，AI 主动执行）：

```
查询顺序（优先级）：

1️⃣ 功能定义层
   └─ docs/01_kb_系统功能详解.md
      ✓ 确认问题涉及的功能是否在系统规划内
      ✓ 查看功能的核心流程图

2️⃣ 进度与状态层
   └─ docs/02_ref_项目进度与变更记录.md
      ✓ 确认功能是否已实现（W1-W8）
      ✓ 查看版本变更历史

3️⃣ 验收标准层
   └─ docs/02_ref_w{X}_验收报告.md（对应阶段）
      ✓ 查看"⚠️ 缺失的功能"部分
      ✓ 查看"👉 已知限制"部分
      ✓ 判断是否属于功能缺失

4️⃣ 排查手册层
   └─ docs/11_module_功能问题排查手册.md
      ✓ 查看"功能不工作"部分的排查步骤
      ✓ 按检查清单逐项排查

5️⃣ Issue 文档层
   └─ docs/06_issue_*.md（所有问题文档）
      ✓ 搜索关键词
      ✓ 查看是否有相同 Issue 已报告
```

**AI 输出示例**：

```
📋 我正在查阅知识库，请稍候...

✓ Step 1: 查 01_kb_系统功能详解.md
  → 结论：模板字段解析是 W2 的核心功能

✓ Step 2: 查 02_ref_项目进度与变更记录.md
  → 结论：W2 已完成，W8 进行中

✓ Step 3: 查 02_ref_w2_验收报告.md
  → 结论：不支持书签、内容控制等高级格式

✓ Step 4: 查 11_module_功能问题排查手册.md
  → 查阅"字段识别问题"的排查步骤

✓ Step 5: 查 06_issue_*.md
  → 发现相关 Issue：06_issue_w8_模板解析精确识别.md
```

### Step 3: 问题诊断与分类（自动判断）

**AI 自动判断问题属于哪一类**：

| 诊断结果 | 判断依据 | 处理方式 |
|---------|---------|---------|
| 📌 已知限制 | 在验收报告的"缺失功能"中找到 | 告知规避方案 + 链接 Issue |
| 📌 已报告问题 | 在 06_issue_*.md 中找到相同问题 | 提供详细的改进方案 |
| 📌 使用错误 | 在排查手册的检查清单中找到原因 | 提供修复指导 |
| 📌 新发现 Bug | 上述三个都查不到 | 创建新的 Issue 文档 |

**AI 输出格式**：

```
🔍 诊断结果
─────────────

【问题分类】新发现 Bug
【优先级】P0（阻断性）
【模块】模板解析（Tools/TemplateParser.cs）

【诊断过程】
✓ 不属于功能规划外项 → 01_kb 中有明确定义
✓ 不属于已知限制 → W2 验收报告中无此限制
✓ 不属于使用错误 → 排查手册检查无异常
✗ 属于新发现问题 → 需要创建 Issue

【根本原因】
模板中的占位符被跨越多个 Run 对象分割时，正则表达式无法识别。
```

## 如何触发这个工作流

### 触发方式1：用户主动说出问题

**用户输入示例**：
```
"我遇到了一个问题，模板上传后字段无法识别"
```

**AI 自动行为**：
- ✓ 自动追问缺失的信息
- ✓ 自动查阅知识索引
- ✓ ✓ 自动诊断问题类型
- ✓ 自动输出结构化报告

### 触发方式2：用户明确要求按流程处理

**用户输入示例**：
```
"按照 03_guide 中的第 7 章流程处理这个问题：[问题描述]"
```

**AI 自动行为**：
- ✓ 立即进入问题处理工作流
- ✓ 逐步执行所有 5 个步骤

### 触发方式3：AI 主动识别

**场景**：用户描述中包含关键词时

**关键词触发列表**：
- "无法识别"、"识别不了"、"不支持"
- "报错"、"异常"、"错误"、"失败"
- "不工作"、"不能用"、"坏了"
- "Bug"、"问题"、"有个事儿"

## 工作流配置细节

### 配置1：问题收集表单

当用户信息不完整时，AI 自动生成表单。表单字段来自：
- `docs/03_guide_ai协作与项目理解指南.md` → 第 7.1 节

### 配置2：知识库查询顺序

查询优先级（不可变更）：
```
01_kb_系统功能详解.md (功能定义)
  ↓
02_ref_项目进度与变更记录.md (实现进度)
  ↓
W*_验收报告.md (功能限制)
  ↓
11_module_功能问题排查手册.md (排查步骤)
  ↓
06_issue_*.md (已报告问题)
```

### 配置3：诊断分类树

诊断结果必须属于以下四类之一：
- 已知限制 (Knwon Limitation)
- 已报告问题 (Reported Issue)
- 使用错误 (User Error)
- 新发现问题 (New Bug)

### 配置4：输出格式规范

所有输出必须遵循：
- 使用 Markdown 格式
- 包含【诊断结果】、【根本原因】、【建议行动】
- 链接相关文档
- 避免输出过长（超过 3000 字时分页）

## AI 自检清单

在执行每一步前，AI 需要自检：

```
执行前清单（Step 1）：
  [ ] 问题信息是否完整（5 个字段都收集到）
  [ ] 是否有具体的重现步骤
  [ ] 是否有测试数据样本（如果需要）

执行前清单（Step 2）：
  [ ] 是否已打开 01_kb_系统功能详解.md
  [ ] 是否已查看相关的 W*_验收报告.md
  [ ] 是否已查看 06_issue_*.md 中的关键词搜索

执行前清单（Step 3）：
  [ ] 是否根据 3 个检查点进行了诊断
  [ ] 诊断结果是否明确属于某一类
  [ ] 是否有充分的证据支持诊断

执行前清单（Step 4）：
  [ ] 是否选择了正确的处理模板
  [ ] 是否包含了具体的可行建议
  [ ] 是否提供了文档链接
```
