## 🚀 AI 第一次进入项目时的做法

### Step 1: 读这 3 个文件（15 分钟）

**优先级顺序**：

1. **README.md**（3 分钟）
   - 快速了解项目是什么
   - 看快速启动方式
   - 知道文档在哪

2. **docs/03_guide_ai协作与项目理解指南.md**（5 分钟）
   - 理解项目的业务链路
   - 学会怎样查找信息
   - 知道代码结构

3. **.instructions.md** 或 **.copilot-instructions.md**（7 分钟）
   - 学会项目的问题处理流程
   - 记住查文档的顺序
   - 了解输出格式规范

完成这 15 分钟后，你就可以开始处理问题了。

### Step 2: 用户给你任务时，按这个流程

当用户说"我遇到一个问题"时：

```
【第1步】收集信息 (2 分钟)
  └─ 追问：影响范围、现象、重现步骤、环境、已尝试
     如果信息不足，自动生成表单

【第2步】查知识索引 (5 分钟)
  ├─ 01_kb_系统功能详解.md → 功能定义
  ├─ 02_ref_项目进度与变更记录.md → 实现状态
  ├─ W*_验收报告.md → 已知限制
  ├─ 11_module_功能问题排查手册.md → 排查步骤
  └─ 06_issue_*.md → 已报告问题

【第3步】诊断分类 (3 分钟)
  └─ 判断属于：已知限制 / 已报告 / 使用错误 / 新Bug

【第4步】输出报告 (2 分钟)
  └─ 【诊断结果】【根本原因】【建议行动】【相关文档】
```

总耗时：**12 分钟**，之后会越来越快。

## 📋 用户最常见的 5 种问题及处理方式

### 问题类型1：功能支持询问

**用户问**：
> "这个功能支持吗？"、"能不能做 XXX"

**AI 处理**（3 步）：
```
1. 查 01_kb_系统功能详解.md 确认功能定义
2. 查 02_ref_项目进度与变更记录.md 看实现状态
3. 回答：支持 + 链接文档
```

**回复示例**：
```
✓ 支持该功能

功能：AI 对话填充
状态：W4 已完成
实现详情：docs/02_ref_w4_实现流程.md
验收标准：docs/02_ref_w4_验收报告.md
```

### 问题类型2：功能不工作

**用户问**：
> "这不工作"、"这个功能坏了"、"为什么不能用"

**AI 处理**（5 步）：
```
1. 追问：具体表现是什么
2. 查 W*_验收报告.md 看已知限制
3. 查 11_module_功能问题排查手册.md 看排查步骤
4. 判断：是已知限制还是使用错误
5. 回答：规避方案或排查指导
```

**判断决策树**：
```
不工作
  ├─ 在"已知限制"中 → 告知规避方案
  ├─ 在排查手册中 → 按步骤指导
  └─ 都找不到 → 创建新 Issue
```

### 问题类型3：报错或异常

**用户问**：
> "报错了"、"这个异常是什么"、"怎样修复"

**AI 处理**（5 步）：
```
1. 收集：完整错误信息、日志、环境
2. 查 11_module_功能问题排查手册.md
3. 查 06_issue_*.md 搜索相同错误
4. 判断：是已知问题还是新问题
5. 回答：排查步骤或 Issue 链接
```

### 问题类型4：功能改进建议

**用户问**：
> "能不能改进 XXX"、"怎样优化"、"有没有更好的方案"

**AI 处理**（4 步）：
```
1. 查 06_issue_*.md 看是否已有改进计划
2. 查 02_ref_项目进度与变更记录.md 看优先级
3. 判断：可行性、优先级、工作量
4. 回答：改进方案或指向 Issue 文档
```

### 问题类型5：代码理解或修改请求

**用户问**：
> "这段代码怎么样"、"帮我改这个"、"为什么这样写"

**AI 处理**（3 步）：
```
1. 先查相关的实现流程文档（W*_实现流程.md）
2. 理解设计意图和约束条件
3. 然后才进行代码分析或建议修改
```

## 🧠 W8 方法论补充：模板解析与 AI 双轨比对

> 适用场景：用户反馈“模板字段没识别到”“表格参数丢了”“`{{}}` 不生效”“`{Acronym.WordAcronym}` 识别失败”

### 方法论1：占位符语法兼容（先识别，再规范）

当用户上传模板时，AI 要先确认系统应支持以下写法：

1. `{字段名}`
2. `{{字段名}}`
3. `[字段名]`
4. `(字段名)`

处理原则：

- 不因语法差异直接判失败。
- 先识别原始语法，再规范化为统一内部格式（建议 `{字段名}`）。
- 规范化行为写入 `warnings`，便于前端提示用户。

### 方法论2：点号字段识别（至少两段，不再只限两段）

遇到点号字段时，不要只按“必须两段”处理。

判定规则：

1. 只要占位符包含 `.` 且段数 >= 2，即进入“表格/复合字段候选”。
2. 两段写法：`{A.B}` -> 表名 `A`，列名 `B`。
3. 多段写法：`{A.B.C}` -> 表名 `A`，列路径 `B.C`。
4. 典型兼容目标：`{Acronym.WordAcronym}` 必须可识别，不可静默丢弃。

### 方法论3：双轨解析（本地规则为主，AI 语义为辅）

模板上传解析建议按这个顺序：

```
本地解析 -> AI 语义解析 -> 差异比对 -> warnings 输出 -> 保存（以本地解析为准）
```

说明：

- AI 比对用于“补充发现”和“风险提示”，不是直接覆盖配置。
- AI 不可用、超时、返回异常都不能阻断上传。

### 方法论4：差异输出标准（统一写 warning）

至少输出这 4 类 warning：

1. 语法规范化提示（例如 `{{项目名称}}` 已规范为 `{项目名称}`）
2. 本地有 AI 无（可能 AI 漏识别）
3. AI 有本地无（可能本地规则漏解析）
4. 点号层级冲突（例如 `A.B` 与 `A.B.C`）

### 方法论5：用户沟通模板（可直接复用）

当用户问“为什么没识别到”时，按这个结构回复：

```
【诊断结果】
当前模板包含扩展占位符/点号字段，已触发兼容解析策略。

【根本原因】
原解析规则偏向标准 {} 与两段点号字段，导致部分写法漏识别。

【建议行动】
1. 启用 AI 双轨比对（aiVerify=true）
2. 将 warnings 中的规范化提示回显给用户确认
3. 对 {Acronym.WordAcronym} 这类字段按表格候选处理

【相关文档】
- docs/06_issue_w8_模板上传AI双轨解析比对.md
- docs/05_api_接口规范.md
```

## ✨ AI 处理问题的最佳实践

### ✓ 做这些

✓ **查文档**：问题有 80% 的可能在文档中有答案  
✓ **链接文档**：每个建议都要附带相关文档链接  
✓ **结构化输出**：用清晰的【】格式组织答案  
✓ **给具体方案**：不要说"可能是..."，要说"根据 XXX 文档..."  
✓ **分页输出**：答案太长时分页，每页 1000-1500 字  

### ✗ 不要做这些

✗ **凭直觉**：不要没查文档就给建议  
✗ **模糊答案**：不要说"可能"、"大概"、"也许"  
✗ **没有依据**：不要给结论而不给文档支持  
✗ **过长输出**：答案太长用户会看不全  
✗ **忽视上下文**：不要忘记 AI 协作指南中的原则

## 🎯 成功处理问题的标志

当 AI 给出的答案同时满足以下条件时，说明处理得很好：

- ✅ 有明确的诊断结论（已知限制 / 已报告 / 使用错误 / 新Bug）
- ✅ 给出了根本原因分析
- ✅ 提供了可立即采取的建议或行动
- ✅ 链接了 2-3 个相关文档
- ✅ 如果是新问题，已创建 Issue 文档
- ✅ 用户看了答案后知道下一步该做什么

```

```
