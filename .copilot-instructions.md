# VS Code Copilot 工作流触发规则（.copilot-instructions.md）

> 这个文件定义了 VS Code GitHub Copilot 在此项目中应该遵循的规则和工作流

---

## 🎯 核心原则

1. **按文档优先**：遇到问题时，先查文档，再看代码
2. **系统化处理**：按预定流程处理，不凭直觉猜测
3. **结构化输出**：所有答案都要有清晰的逻辑结构
4. **链接相关文档**：每个答案都要附带相关文档链接

---

## 🚀 问题处理工作流（当用户报告问题时自动触发）

```
用户报告问题
  │
  ├→ [Step 1] 收集信息（如果不完整，自动追问）
  │   └─ 追问：影响范围、现象、重现步骤、环境、已尝试
  │
  ├→ [Step 2] 查阅知识索引（按优先级自动查）
  │   ├─ 01_kb_系统功能详解.md → 确认功能定义
  │   ├─ 02_ref_项目进度与变更记录.md → 确认实现状态
  │   ├─ W*_验收报告.md → 查看已知限制
  │   ├─ 11_module_功能问题排查手册.md → 查看排查步骤
  │   └─ 06_issue_*.md → 搜索相同问题
  │
  ├→ [Step 3] 问题诊断与分类
  │   ├─ 是否已知限制？ → 建议规避方案
  │   ├─ 是否已报告问题？ → 链接 Issue 文档
  │   ├─ 是否使用错误？ → 指导排查
  │   └─ 是否新 Bug？ → 创建 Issue 文档
  │
  └→ [Step 4] 输出结构化报告
      └─ 【诊断结果】【根本原因】【建议行动】【相关文档】
```

## ⚡ 快速触发清单

### 问题关键词（自动进入问题处理工作流）

当用户的输入包含以下任何关键词时，**主动进入问题处理模式**：

**功能性问题**：
- "无法识别"、"识别不了"、"没识别出来"
- "不支持"、"不能用"、"不工作"
- "报错"、"异常"、"错误"、"失败"
- "问题"、"Bug"、"缺陷"

**状态询问**：
- "怎么样"、"支持吗"、"能不能"
- "已完成吗"、"做好了吗"、"什么时候"

**改进诉求**：
- "能否改进"、"怎样优化"、"有没有更好的"

### AI 自动行为规则

| 用户输入 | 自动检测 | AI 行为 |
|--------|--------|--------|
| "这个功能支持吗？" | 功能询问 | 查 01_kb 和 02_ref 回答 |
| "这个不工作" | 功能问题 | 进入问题处理工作流 |
| "我遇到个问题" | 问题上报 | 收集信息 → 查文档 → 诊断 |
| "这是 Bug 吗？" | 问题判定 | 按步骤诊断，给出结论 |

## 📚 文档查询速记

### 查阅优先级（固定顺序）

```
1️⃣ 功能定义层 → docs/01_kb_系统功能详解.md
   问：功能是否在系统规划内？
   何时用：判断功能是否应该支持
   
2️⃣ 进度状态层 → docs/02_ref_项目进度与变更记录.md
   问：功能在哪个阶段？已完成吗？
   何时用：判断功能是否已实现
   
3️⃣ 验收标准层 → docs/02_ref_w{N}_验收报告.md
   问：功能的限制是什么？
   何时用：判断是否属于已知限制
   
4️⃣ 排查指南层 → docs/11_module_功能问题排查手册.md
   问：怎样排查这个问题？
   何时用：指导用户自助排查
   
5️⃣ 问题追踪层 → docs/06_issue_*.md
   问：是否已有相同问题的分析？
   何时用：搜索关键词找已报告问题
```

### 快速查询方法

当需要查文档时：

```
场景1：用户问"这功能支持吗"
  → 直接查 01_kb_系统功能详解.md
  → 在功能列表中找到对应功能
  → 回答支持情况

场景2：用户说"这个不工作"
  → 先查 02_ref_项目进度与变更记录.md 看状态
  → 再查对应的 W*_验收报告.md 看已知限制
  → 最后查 11_module_功能问题排查手册.md
  → 按排查步骤指导用户

场景3：用户说"这是个 Bug"
  → 查 06_issue_*.md 搜索关键词
  → 有相同 Issue → 链接文档
  → 无相同 Issue → 创建新 Issue
```

## 📋 问题分类与处理模板

### 分类1：已知限制

**识别方式**：在 W*_验收报告 的"⚠️ 缺失的功能"中找到

**处理方式**：
```
【诊断结果】已知限制 ✓
【功能状态】不支持（W{N} 设计决策）
【规避方案】[具体规避步骤]
【改进计划】[相关 Issue 链接]
```

### 分类2：已报告问题

**识别方式**：在 06_issue_*.md 中搜索关键词找到相同问题

**处理方式**：
```
【诊断结果】已报告问题 ✓
【问题文档】[Issue 文件名和链接]
【当前状态】[待实施 / 进行中 / 已修复]
【改进方案】[从 Issue 文档提取的改进方案]
【临时规避】[从 Issue 文档提取的规避方案]
```

### 分类3：使用错误

**识别方式**：在排查手册的检查清单中找到问题原因

**处理方式**：
```
【诊断结果】使用错误 ✓
【根本原因】[找到的具体原因]
【解决步骤】
  1. [检查项1]
  2. [检查项2]
  3. [修复操作]
【验证方法】[如何验证已解决]
```

### 分类4：新发现问题

**识别方式**：前 3 种都找不到

**处理方式**：
```
【诊断结果】新发现问题 ✨
【问题模块】[受影响的模块]
【现象描述】[具体表现]
【根本原因】[初步分析]
【后续行动】我已创建 Issue 文档：
  📄 docs/06_issue_w8_问题描述.md
  该文档包含详细的分析方案和实施计划
```

## 🔍 输出格式规范

### 所有答案必须包含

✓ **问题诊断**：清晰的分类结果  
✓ **根本原因**：不能模糊，要有具体分析  
✓ **相关文档**：至少链接 1-2 个相关文档  
✓ **可行建议**：用户能立即采取的行动  
✓ **后续路径**：如果问题需要改进，给出预期周期  

### 不允许的输出

✗ 仅列问题不给方案  
✗ 没有文档支持的结论  
✗ 模糊的"可能是"或"也许"  
✗ 过长的输出（>3000字时需分页）  
✗ 没有链接相关文档

### 场景2：用户说"这不工作"

```
快速处理流程：
1. 查 W*_验收报告.md 看已知限制
2. 查排查手册看排查步骤
3. 给出排查指导或问题分类

```

**输出示例**：

✓ 根本原因分析

## 📌 工作流触发条件

### 主动进入问题处理流程的条件

✓ 用户明确说有问题  
✓ 用户询问功能支持情况  
✓ 用户报告错误或异常  
✓ 用户问"怎样"或"能否"等改进问题  

### 不需要进入问题处理流程的场景

✗ 代码审视请求  
✗ 代码重构请求  
✗ 一般技术问题（与项目无关）  
✗ 文档更新请求  

---

## ⚠️ 常见错误（避免）

| ❌ 错误做法 | ✅ 正确做法 |
|-----------|-----------|
| "这可能是 Bug，你试试改 X 代码" | "让我按流程诊断后再给建议" |
| "我不知道，自己看代码吧" | "查文档后给出诊断结论" |
| "可能是权限问题，或者环境问题" | "按排查手册逐项排查" |
| "这是个新功能，时间不够" | "创建 Issue 文档详细规划" |
| 输出 5000+ 字的长篇幅 | 分页输出，每页 1000-1500 字 |
